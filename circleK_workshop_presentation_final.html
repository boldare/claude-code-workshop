LLM

Czym jest LLM?

LLM (Large Language Model)

Model językowy, który przewiduje kolejny fragment tekstu (token) na podstawie kontekstu. Jak T9 w starym telefonie, ale z ogromną wiedzą o języku, kodzie i logice.

Działanie probabilistyczne

Model oblicza prawdopodobieństwo każdego możliwego następnego tokenu i wybiera. Ten sam prompt może dać różne odpowiedzi.

Nie "wiedzą" — szacują

Gdy model nie ma dobrego kandydata, "zgaduje" — stąd halucynacje. Zawsze weryfikuj wyniki.

Tokenizacja

Token to podstawowa jednostka tekstu dla modelu — nie zawsze równa się jednemu słowu. Tokenizer rozbija tekst na fragmenty ze swojego słownika (~100 000 tokenów):

Angielski
"authentication" → 1 token
(częste angielskie słowo)
Polski (+70% tokenów!)
"uwierzytelniania" → 4 tokeny
(polskie słowo rozbijane na kawałki)
JęzykTekstTokeny
EN "Create a user authentication service" ~7 tokenów
PL "Stwórz serwis uwierzytelniania użytkowników" ~12 tokenów (+70%!)
Tokeny to waluta LLM — za nie płacisz i one definiują limity.
LLM

Modele dostępne w Anthropic / Claude Code

Model Charakterystyka Knowledge cutoff Input / Output (za 1M tokenów) Kiedy używać
Claude Opus 4.6 Najbardziej zaawansowany, głęboki reasoning Maj 2025 $5 / $25 Planowanie architektury, złożony debugging, code review
Claude Sonnet 4.6 domyślny Złoty środek: jakość + szybkość Sierpień 2025 $3 / $15 Codzienna praca: coding, testy, refactoring — 80% zadań
Claude Haiku 4.5 Najszybszy i najtańszy Luty 2025 $1 / $5 Boilerplate, CRUD, szybkie pytania
Output jest ~5× droższy niż input — odpowiedź modelu kosztuje znacznie więcej niż prompt. Dlatego wybór modelu i kontrola długości odpowiedzi mają bezpośredni wpływ na koszty.

Kontekst nadpisuje wiedzę wytrenowaną

To co podamy w oknie kontekstowym ma priorytet nad wiedzą z treningu. Jeśli piszesz, że funkcja login() rzuca TokenExpiredException — model użyje tej informacji zamiast własnych założeń. Stąd siła CLAUDE.md i systemu promptów.

Pułapki okna kontekstowego

Przepełnienie

Gdy okno się zapełnia, Claude Code automatycznie kompresuje wcześniejsze wiadomości (/compact). Tracisz część informacji = model może halucynować brakujące szczegóły.

"Zagubiony środek"

Model najlepiej pamięta początek (system prompt) i koniec (ostatnie wiadomości). Informacje ze środka długiej konwersacji mają tendencję do "znikania".

Podejście agentowe — Agentic Loop

Claude Code nie generuje jednorazowej odpowiedzi — działa w pętli agentowej, iteracyjnie zbliżając się do rozwiązania. To fundamentalna różnica względem Copilota czy czata.

Gotowe!
1. Odbierz zadanie

Użytkownik opisuje co chce osiągnąć (np. "napraw błąd w logowaniu")

2. Zaplanuj i zdecyduj

Model analizuje kontekst: czyta pliki, przegląda strukturę projektu, planuje podejście

3. Użyj narzędzia / Działaj

Model wywołuje narzędzie: edytuje plik, uruchamia komendę, szuka w kodzie

4. Obserwuj wynik

Model analizuje rezultat: czy test przeszedł? Czy kod się kompiluje? Czy jest błąd?

5. Iterować?

Jeśli cel nie osiągnięty = powrót do kroku 2 z nową wiedzą; jeśli tak = zakończ

Kluczowa cecha: Claude Code sam decyduje, ile iteracji potrzebuje. Pojedyncze polecenie może wywołać dziesiątki cykli — np. przeczytanie 5 plików, edycja 3, uruchomienie testów, naprawa błędów, ponowne uruchomienie testów.
Techniki promptowania

Iteracyjne promptowanie — zacznij prosto, doprecyzowuj

Nie musisz pisać idealnego promptu. Buduj kontekst krok po kroku:

Krok 1: "Przejrzyj moduł auth i powiedz co można poprawić"
→ Claude daje listę sugestii
Krok 2: "Skupmy się na punkcie 3 — token refresh. Jak to zrefaktorować?"
→ Claude proponuje podejście
Krok 3: "OK, zaimplementuj. Napisz testy. Upewnij się, że stare przechodzą."
→ Claude implementuje i testuje
💬

Claude ma historię rozmowy w oknie kontekstowym — każdy prompt buduje na poprzednich.

Techniki promptowania

Scope narrowing — zawężaj zakres

ŹLE
"Znajdź i napraw bug z timeout"
→ Claude czyta 50 plików, kontekst pełny
DOBRZE
"Znajdź i napraw bug z timeout w src/api/gateway/.
Nie dotykaj innych modułów."
→ Claude czyta 5 plików, szybko trafia w sedno

Im bardziej zawęzisz zakres, tym szybciej i taniej Claude znajdzie rozwiązanie.

Hands-on

Wypróbuj scope narrowing na plikach prd.md i tech-stack.md:

Krok 1 — szeroki prompt (obserwuj ile Claude czyta i jak ogólna jest odpowiedź):

"Przejrzyj projekt i powiedz co wiesz"

Krok 2 — zawężony prompt (obserwuj precyzję i szybkość):

"Na podstawie prd.md wskaż tylko wymagania dotyczące [wybierz jedną funkcję]. Nie analizuj reszty."

Self-Criticism (Samokrytyka)

Poproś Claude o weryfikację własnej odpowiedzi przed zakończeniem:

Zanim skończysz, zweryfikuj swoją odpowiedź:

- Czy nie ma race conditions?

Jeśli znajdziesz błędy, napraw je.

Hands-on

Wybierz jedną funkcję z prd.md i użyj metody sokratejskiej:

Krok 1 — zainicjuj wywiad:

"Przeczytaj prd.md. Chcę zaimplementować [wybierz jedną funkcję]. Przeprowadź ze mną wywiad używając AskUserQuestion — pytaj o edge case'y i tradeoffs, o których mógłbym nie pomyśleć. Na końcu podsumuj ustalenia."

Obserwuj: Claude pyta zamiast zgadywać → odpowiedzi ujawniają luki w wymaganiach.

Krok 2 — porównaj:

"Zaimplementuj [tę samą funkcję] na podstawie prd.md" — i porównaj jakość wyniku.

Few-Shot Prompting

Podaj 3-5 zróżnicowanych przykładów w tagach <examples>. Szczególnie przydatne w CLAUDE.md i skills:

<examples>

Adaptive Thinking w Claude

Adaptive Thinking — inteligentne rozumowanie

Claude Opus 4.6 i Sonnet 4.6 wspierają Adaptive Thinking — model dynamicznie decyduje kiedy i ile "myśleć" przed odpowiedzią. W przeciwieństwie do fixed-budget thinking, automatycznie dopasowuje głębokość rozumowania do złożoności zadania.

PoziomZachowanieKiedy używać
lowMinimalne myślenie, pomija proste problemySzybkie pytania, boilerplate, latency-sensitive
mediumUmiarkowane myślenieWiększość codziennych zadań — zalecany start
high (domyślny)Głębokie rozumowanieZłożony debugging, architektura, multi-step
Konfiguracja
/model + strzałki → effort
CLAUDE_CODE_EFFORT_LEVEL=low|medium|high
"ultrathink" → wymusza high effort
Option+T / Alt+T → włącz/wyłącz
Interleaved Thinking

Claude myśli między wywołaniami narzędzi — analizuje wyniki pośrednich kroków. Kluczowe dla agentowych workflow.

Ważne: Tokeny myślenia są naliczane (nawet jeśli Claude 4 pokazuje streszczenie myślenia, płaci się za pełne tokeny). Kontroluj koszty przez max_tokens i effort.

Reguły warunkowe — ładuj tylko gdy potrzebne

Domyślnie każdy plik w rules/ ładuje się zawsze — zużywa kontekst nawet gdy niepotrzebny. Dodaj paths: aby reguła wchodziła tylko przy pracy z pasującymi plikami:

Plan Mode: workflow

Plan Mode — bezpieczna analiza i planowanie

Plan Mode to tryb, w którym Claude czyta i analizuje kod, ale nie może modyfikować plików ani uruchamiać komend.

Kiedy używać?

Multi-step implementation — feature wymagający edycji wielu plików

Eksploracja kodu — rozumiej codebase zanim cokolwiek zmienisz

Interaktywne planowanie — iterujesz nad kierunkiem z Claude

Code review — analiza bez ryzyka przypadkowej modyfikacji

Typowe błędy

Typowe błędy i jak ich unikać

1. "Sesja-śmietnisko" (Kitchen Sink Session)

Mieszanie niepowiązanych zadań w jednej sesji. Kontekst pełny śmieci.

Rozwiązanie: /clear między zadaniami.

2. Niekończące się korekty

Claude robi błąd, poprawiasz, dalej błąd… Kontekst zatruwany nieudanymi próbami.

Rozwiązanie: Po 2 nieudanych korektach — /clear + lepszy prompt.

3. Przeładowany CLAUDE.md

Za długi plik = Claude może ignorować część instrukcji.

Rozwiązanie: Przenieś szczegółowe reguły do .claude/rules/.

4. Brak weryfikacji

Claude generuje kod, który wygląda ok, ale nie działa na edge case'ach.

Rozwiązanie: Zawsze dostarczaj testy, skrypty walidacyjne, screenshoty.

5. Nieskończona eksploracja

"Zbadaj to" bez zakresu = Claude czyta setki plików, kontekst pełny.

Rozwiązanie: Zawężaj zakres lub używaj subagentów.
