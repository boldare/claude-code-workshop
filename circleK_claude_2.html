Dobry scenariusz

Trzy poziomy szczegółowości

Za prosty

Given a movie is showing
When I book a seat
Then the booking
  is confirmed

Nie weryfikuje niczego konkretnego. Przejdzie nawet gdy logika jest zepsuta.

Odpowiedni ✓

Given screening "Inception"
  has seat A1 AVAILABLE
When Alice selects seat A1
Then reservation is PENDING
And seat A1 becomes LOCKED

Konkretne wartości, dwa stany w Then, czytelny dla biznesu.

Za szczegółowy

When POST /reservations
  body: {"seatIds":[42]}
Then HTTP 201
And SELECT status FROM
  seats WHERE id=42
  = 'LOCKED'

Implementacja wyciekła do scenariusza. Zmiana endpointu = zmiana scenariusza.

Wskazówka: Then powinno opisywać efekt z perspektywy użytkownika — co widzi, co dostaje, jaki status ma jego rezerwacja. Nie: co jest w bazie danych ani jaki kod HTTP zwrócił serwer.

Jak wygląda test

Jak wygląda test Playwright

Test
test('customer selects available seat', async ({ page }) => {
  await page.goto('/screenings/1');

  const seat = page.getByTestId('seat-A1');
  await expect(seat).toHaveAttribute('data-status', 'AVAILABLE');

  await seat.click();
  await expect(seat).toHaveAttribute('data-status', 'LOCKED');
  await expect(page.getByRole('button', { name: 'Potwierdź rezerwację' }))
    .toBeVisible();
});
Co tu widać?
  • goto — otwiera stronę w prawdziwej przeglądarce
  • getByTestId / getByRole — lokatory semantyczne, nie CSS
  • expect(...).toHaveAttribute — asercja na stan
  • Brak sleep — Playwright sam czeka na elementy

Definicja i spektrum

Czym jest (a czym nie jest) VibeCoding?

Definicja rdzeniowa: opisujesz co chcesz w języku naturalnym, AI generuje kod, celowo nie czytasz ani nie rozumiesz kodu.

"If an LLM wrote every line of your code, but you've reviewed, tested, and understood it all — that's NOT vibe coding. That's using an LLM as a typing assistant."
— Simon Willison (autor deta, były dyrektor PSF)
VibeCoding AI-assisted engineering
Rozumienie kodu Celowo pominięte Wymagane
Przegląd wyników Opcjonalny Obowiązkowy
Kto to robi Każdy Ekspert + AI
Główna wartość Eksploracja, prędkość Jakość + prędkość
Ryzyko Wysokie bez weryfikacji Niskie

Spektrum: od pure vibe (niedewelopment, zero code review) → do full AI-assisted engineering (ekspert weryfikuje każdą linię).

To nie jest pytanie "czy używasz AI" — to pytanie o świadomą postawę wobec kontroli nad kodem.

Case study: ETH2030

Problem: 65 planowanych upgrades Ethereum rozsianych po dziesiątkach EIPów, wiele w draft. Nikt nie sprawdził czy architektonicznie do siebie pasują. Ryzyko: client teams mogłyby odkryć w 2030, po latach produkcyjnego inżynierowania, że fundamentalne założenia nie trzymają się kupy.

Rozwiązanie: Claude Code Opus 4.6 · 6 dni · $5,750 · 2,77 mld tokenów → 702k linii Go · 36 126 testów zdanych · sync z mainnetem

"All 65 items fit into a single codebase. The types align. The interfaces connect. That is more than most five-year technology roadmaps achieve before anyone writes a line of code." — ETH2030 team

Vitalik Buterin w odpowiedzi: blog software napisany w 1 GODZINĘ na własnym laptopie (gpt-oss:20b). "What matters is where the trend is going."

Paradoks weryfikacji (ETH2030)

"The gap between 'ValidateAttestation() returns nil' and 'survives a reorg attack during peak MEV' is measured in years of engineering."

Pure-Go BLS crypto jest 10–100x wolniejszy niż produkcyjny C/Rust. Walidator przegapiałby attestation deadlines o rząd wielkości. Architecturally correct, operationally broken.

Paradoks kwalifikacji

METR study: doświadczeni deweloperzy byli 19% WOLNIEJSI z AI niż bez (przy własnej prognozie: +24%).

AI działa najlepiej dla ekspertów, najgorzej dla nowicjuszy — odwrotnie niż większość zakłada. Nowicjusz nie rozpozna niebezpiecznego kodu wygenerowanego przez AI.

Świadomy wybór narzędzia

VibeCoding ma sens gdy

  • Prototyp / proof-of-concept / MVP
  • Eksploracja — "czy to w ogóle jest możliwe?"
  • Narzędzia wewnętrzne, side projekty, skrypty
  • Domenowa wiedza pozwala zweryfikować output
  • Koszt błędu jest niski

VibeCoding jest ryzykowny gdy

  • System produkcyjny z danymi użytkowników
  • Wymagania security/compliance (finanse, medycyna)
  • Codebase do wieloletniego maintenance
  • Brak testów i feedback loops
  • Architektura, którą ktoś będzie debugować za rok

Zasady inżynierskie, które stają się WAŻNIEJSZE przy VibeCoding, nie mniej:

Fast feedback loops

(testy, CI/CD) — bez nich prędkość = niewidzialne ryzyko

Modularity

ogranicza blast radius i umożliwia równoległą pracę agentów

Systematyczna weryfikacja

nie "uruchomił się = działa"

Tracer bullet approach

zamiast big bang, iteracyjne potwierdzanie że kierunek jest dobry

"A reference implementation generates not production-quality code, but production-quality questions." — ETH2030 team

Vitalik Buterin o strategii: "Take half the gains from AI in speed, and half the gains in security: generate more test-cases, formally verify everything, make more multi-implementations."

VibeCoding jest narzędziem, nie religią. Świadomy wybór kiedy "give in to the vibes" — to właśnie jest nowa umiejętność eksperta.

Ograniczenia

Dlaczego to nie wystarczy?

Statyczna dokumentacja ma datę ważności

  • modules.md i [moduł]-analysis.md to snapshot — kod idzie dalej, dokumentacja zostaje
  • Nowy event handler dodany tydzień temu? Nie ma go w analizie
  • Nikt nie robi code review pliku analysis.md — błędy są ciche

AI ufa analizie za bardzo

  • Przeczyta plik → uzna że to komplet zależności → przestanie szukać
  • Częściowa mapa jest gorsza niż brak mapy — bo brak mapy wymusza eksplorację
  • Paradoks: im lepsza analiza, tym bardziej AI się na niej opiera

Koszt rośnie, wartość maleje

  • 20 min/moduł × 30 modułów = 10+ godzin jednorazowej pracy
  • Po 2 sprintach połowa analizy jest nieaktualna
  • Kto zaktualizuje? Nikt — bo to nie jest kod, który się testuje
